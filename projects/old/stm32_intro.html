For the code we will make a simple LED on program when pressing a button. PA5 is our input which will be connected to a button and PA0 is out output which will be connected to an LED.

Below is some simulation code simulated using Arduino on tinkercad just to iron out the logic of the code.

Arduino Code


const int LED_PIN = 0;
const int BUTTON_PIN = 5;

void setup() {
    pinMode(LED_PIN, OUTPUT);
    pinMode(BUTTON_PIN, INPUT_PULLUP);
}

void loop() {
    if (digitalRead(BUTTON_PIN) == LOW) {
        digitalWrite(LED_PIN, HIGH);
    } else {
        digitalWrite(LED_PIN, LOW);
    }
}

Arduino Simulation

[photo7]

The equivalent STM32 code is below.

STM32 code


while (1)
{
    if (HAL_GPIO_ReadPin(GPIOA, GPIO_PIN_5) == GPIO_PIN_RESET)
    {
        // Button pressed (active LOW)
        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_0, GPIO_PIN_SET);
    }
    else
    {
        HAL_GPIO_WritePin(GPIOA, GPIO_PIN_0, GPIO_PIN_RESET);
    }
}

STM32 simulation

There is no good STM32 simulation tool. The industry standard is to use real hardware to debug.

Closest thing to simulating STM32 is RENODE which is not easy to setup and use.

The DFU (device firmware upgrade) bootloader is stored in the system memory of the STM32 by factory default. In order to upgrade the firmware via usb we need the DFU bootloader to be running. Therefore, to flash software via usb we need to be in the “System memory” boot mode.

Problem:

Usb device not being detected properly

I could get my device to be detected by windows but it couldn’t identify it. See images below

[photo10] [photo11]

We should expect for it to say “USB\VID_0483&PID_DF11” instead of “USB\DEVICE_DESCRIPTOR_FAILURE” this means that windows has detected something is plugged in but wasn’t able to read the description for it.

There are a few reasons why this could be the case

1. Usb cable is power only or faulty

2. D+ / D− hardware issues

3. Missing or incorrect pull-up

I tested the usb cable and was able to transfer a photo from an old phone. So the cable is fine.

Tested the continuity between D+ (PA12) D-(PA11) and they seem to have continuity. So they are all good. I also tested the pullup value of PA12 to 3.3V which is 10kOhm. I think this is incorrect. I think it should be 1.5kohm.

To do these tests I unplugged the STM32 and put my multimeter probes on the points outlined below

This guy [2] looks like he has the same problem as me and his fix was to solder a 1.8k resistor in parallel.

I didn’t have a 1.8kohm resistor but I had a 2kohm which was probably close enough so I soldered that in parallel to get a pulldown resistance of ~1.6kohm. see photo below.

[photo14]

This unfortunately did not fix my problem and the stm32 is still unrecognised by the pc.

I did some more digging and I found that some cheap boards (like this one) did not actually connect boot1 and leaves it floating. My board for some reason had a 100kohm resistor between PB2 and the boot1 jumper. This seemed wrong because it should be tied to ground to get 0v to boot into system memory.

so I removed that 100kohm resistor and shorted the connection hoping that this would help. See below.

[photo15]

Unfortunately again this did not fix my problem. It could be just because it’s a cheap board and doesn’t actually include usb functionality. My next idea is to flash using ST-link instead of usb and hope that that works better.
